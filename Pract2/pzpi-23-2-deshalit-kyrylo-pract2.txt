3 ОПИС ВИКОНАНОЇ РОБОТИ
3.1 Загальні відомості про систему
Назва системи: YouTube
Розробник: Google
Призначення системи: 
YouTube — це вебплатформа для зберігання, обробки та розповсюдження відеоконтенту, яка використовується широким колом користувачів: глядачами, авторами контенту, блогерами, медіакомпаніями та рекламодавцями; система забезпечує такі основні функції, як завантаження та перегляд відео, пошук і рекомендації контенту, підписки на канали, взаємодія через коментарі та оцінки, а також монетизація контенту через рекламу й партнерські програми.
3.2 Архітектурний стиль системи

Архітектура платформи YouTube базується на поєднанні клієнт-серверного підходу, мікросервісної архітектури та використання CDN: клієнт-серверна модель забезпечує взаємодію користувачів (веб або мобільний клієнт) із серверною частиною системи, мікросервіси реалізують основну бізнес-логіку у вигляді незалежних компонентів (обробка відео, пошук, рекомендації, автентифікація), що дозволяє масштабувати їх окремо та підвищує відмовостійкість, а мережі доставки контенту (CDN) відповідають за швидке та ефективне розповсюдження відео через географічно розподілені сервери; такий вибір архітектури обґрунтований потребою обробляти величезні обсяги трафіку, мінімізувати затримки при відтворенні відео та забезпечувати стабільну роботу системи для мільйонів користувачів одночасно.


3.3 Основні компоненти системи
Опис підсистем:
Інтерфейс користувача
Забезпечує взаємодію користувача із системою через веббраузер або мобільний застосунок: відображення відео, сторінок каналів, результатів пошуку, рекомендацій, а також можливість завантаження контенту, залишення коментарів і керування профілем.
Серверна частина
Виконує обробку запитів від клієнтів, маршрутизацію через API та взаємодію між підсистемами; відповідає за автентифікацію, доступ до сервісів, балансування навантаження та передачу даних користувачам.

Рівень бізнес-логіки
Реалізований у вигляді мікросервісів, які виконують основні функції системи: обробка та транскодування відео, формування рекомендацій, пошук, робота з підписками, коментарями та аналітикою; забезпечує правила роботи системи та взаємодію між компонентами.
Рівень зберігання даних
Включає бази даних і розподілені сховища, де зберігаються відеофайли, метадані, інформація про користувачів, історія переглядів і інші дані; оптимізований для роботи з великими обсягами інформації та швидкого доступу.
Допоміжні сервіси
Містять додаткові компоненти, що підтримують роботу системи: системи кешування, CDN для доставки відео, сервіси моніторингу, логування, безпеки, модерації контенту та обробки подій, які забезпечують стабільність, продуктивність і масштабованість платформи.

3.4 Взаємодія компонентів
Опис потоків даних, протоколів взаємодії, механізмів синхронізації.

Рисунок 1.1 – Високорівнева архітектурна діаграма Youtube
Коли користувач запускає YouTube, застосунок спочатку звертається до найближчого CDN/edge-сервера, звідки швидко завантажуються кешовані дані, такі як мініатюри чи частини відео. 
Усі динамічні запити (стрічка, пошук, сторінка перегляду) проходять через API Gateway або балансувальник навантаження, який спрямовує їх до відповідних сервісів. Основні сервіси (Feed, Watch, Search, Upload, Auth) працюють із базами даних та індексами (метадані відео, пошуковий індекс, соціальний граф) і зберігають самі відеофайли в об’єктному сховищі. 
Під час перегляду або завантаження відео події надсилаються в Apache Kafka, де фонові системи виконують транскодування (створення різних якостей відео) та обробку рекомендацій і аналітики. 
Оброблені відео потім доставляються користувачам через CDN для максимальної швидкості, тоді як сторінки продовжують отримувати актуальні дані через сервіси та сховища.

Рисунок 1.2 – Діаграма прийому та збереження інформації
Потік даних починається з клієнта, який надсилає HTTP(S) POST-запит на завантаження відео через API Gateway / Load Balancer; на цьому етапі використовуються REST для передачі даних і автентифікації. 
Далі запит маршрутизується до Video Upload Service, який приймає файл, виконує первинну валідацію та зберігає «сирий» відеофайл у об’єктне сховище, використовуючи внутрішні протоколи доступу до розподіленого storage (часто через HTTP-based API або gRPC). Паралельно формується запис у Metadata DB (метадані відео, статус, ідентифікатори файлів).
Після цього ініціюється обробка відео в Transcoding Service, куди дані передаються або синхронно через внутрішні API (наприклад, gRPC), або асинхронно через черги повідомлень/події (наприклад, з використанням Apache Kafka), що дозволяє відокремити завантаження від ресурсоємної обробки. Transcoding Service генерує кілька версій відео (renditions) у різних форматах і якостях та зберігає їх у тому ж об’єктному сховищі, оновлюючи відповідні записи в Metadata DB. Механізми синхронізації реалізуються через асинхронні події та черги, що забезпечують надійність і повторну обробку у разі збоїв, а також через статуси в базі даних, які відображають стан обробки відео; це дозволяє різним сервісам узгоджено працювати без жорсткої залежності один від одного. Таким чином, система забезпечує масштабовану, відмовостійку та ефективну обробку великих обсягів відеоданих.

Рисунок 1.3 – Діаграма взаємодії сервісів
Сервіс завантаження відео (Video Upload Service)
Компоненти: сервіс завантаження взаємодіє із сервісом зберігання відео та сервісом транскодування. Після завантаження відео воно зберігається як «сирий» контент в об’єктному сховищі (наприклад, Amazon S3). Далі сервіс транскодування обробляє відео, створюючи різні формати та роздільні здатності.
API: POST /upload-video — прийом відеофайлу та метаданих.

Сервіс стрімінгу відео (Video Streaming Service)
Компоненти: після обробки відео цей сервіс відповідає за його доставку користувачам. Він отримує відео з розподіленого сховища та передає його у вигляді потоку (stream), враховуючи можливості пристрою користувача та швидкість інтернету.
API: GET /video/{video_id}/stream — отримання частин відео для потокового відтворення.

Сервіс рекомендацій (Recommendation Service)

Компоненти: система рекомендацій використовує алгоритми машинного навчання, які аналізують поведінку користувача, історію переглядів і метадані відео для формування персоналізованих рекомендацій.
API: GET /recommendations — отримання індивідуальних рекомендацій для користувача.

Сервіс пошуку (Search Service)

Компоненти: забезпечує пошук відео, каналів і плейлистів за ключовими словами. Для цього індексує метадані відео та використовує системи повнотекстового пошуку, такі як Elasticsearch, що дозволяє швидко знаходити результати.
API: GET /search?q={query} — пошук відео за запитом.

3.5 Масштабованість та надійність

Балансування навантаження
YouTube використовує комплексну систему балансування навантаження, щоб рівномірно розподіляти трафік між своїми серверами та компонентами, забезпечуючи швидке та надійне обслуговування користувачів. Це включає:

Глобальні балансувальники навантаження: направляють запити користувачів до найближчого та найменш завантаженого датацентру. NetScalar, наприклад, використовувався для балансування навантаження та кешування статичного контенту 

Локальні балансувальники навантаження: всередині кожного датацентру вони розподіляють трафік між серверами додатків, базами даних та іншими мікросервісами.

Динамічне перенаправлення HTTP: у разі перевантаження сервера в одному кеш-місці клієнт може бути перенаправлений до менш завантаженого сервера в тому ж місці кешу 

Керування трафіком: YouTube використовує складні механізми для маршрутизації відео та інших даних через свою мережу, оптимізуючи продуктивність затримки та запобігаючи перевантаженню.

Відмовостійкість
Надійність є ключовим принципом дизайну системи YouTube. Щоб забезпечити безперебійну доступність послуг, вони застосовують такі стратегії:

Надмірність і реплікація: кожен критично важливий компонент (сервери, бази даних, сховища) має кілька резервних копій або копій. Якщо один компонент виходить з ладу, його роль миттєво бере на себе інший.

Географічна розподіленість: датацентри YouTube розміщені по всьому світу. Це гарантує, що якщо регіональна подія вплине на один датацентр, користувачі все одно зможуть отримати доступ до сервісу з іншого.

Стійкій до помилок дизайн: система розроблена таким чином, щоб продовжувати функціонувати навіть у разі збою окремих компонентів. Це досягається за допомогою таких методів, як ізоляція збоїв та автоматичне відновлення 

Моніторинг та оповіщення: безперервний моніторинг продуктивності системи та автоматичні оповіщення дозволяють швидко виявляти та усувати проблеми, перш ніж вони вплинуть на користувачів

Кешування
Кешування є життєво важливим для швидкої доставки контенту та зменшення навантаження на основні сервери YouTube.

Мережа доставки контенту (CDN): це основа доставки відео YouTube. Копії популярних відео кешуються на периферійних серверах, стратегічно розташованих близько до користувачів по всьому світу. Коли користувач запитує відео, воно подається з найближчого вузла CDN, що різко зменшує затримку та навантаження на сервери 

Кешування на рівні додатків: часто використовувані дані кешуються в оперативній пам'яті на серверах додатків, щоб уникнути постійних запитів до бази даних.
Кешування метаданих: для прискорення доступу до інформації про відео, таку як назви, описи та кількість переглядів, використовується кешування метаданих 

BigTable для мініатюр: YouTube використовує BigTable Google для ефективного зберігання та кешування великої кількості маленьких об'єктів, таких як мініатюри відео. BigTable об'єднує маленькі файли в один блок, що робить його швидким і стійким до збоїв


Контейнеризація або хмарна інфраструктура

YouTube, як частина Google, використовує власну масштабовану хмарну інфраструктуру, яка є однією з найдосконаліших у світі. Вони використовують цілі датацентри по всьому світу, щоб керувати своїми операціями.

Мікросервісна архітектура: YouTube побудовано на архітектурі мікросервісів, що дозволяє незалежно розробляти, розгортати та масштабувати різні частини системи 
Це важливо для управління величезною складністю та масштабом платформи.

Розподілена система: це масово розподілена система, що базується на надійній інфраструктурі Google. Це дозволяє їм справлятися з постійно зростаючим трафіком користувачів та відеоконтентом.

Внутрішні інструменти Google: Google розробив власні передові інструменти та технології, такі як Colossus (розподілене сховище), Spanner (глобально узгоджена база даних) та Bigtable (NoSQL база даних з високою пропускною здатністю), які лежать в основі інфраструктури YouTube. Хоча термін "контейнеризація" не завжди прямо вживається в цьому контексті, інфраструктура Google, безумовно, використовує подібні принципи ізоляції та ефективності ресурсів для розгортання та керування своїми сервісами.

204   <!-- ?? Principle 1 ?? -->
205   <div class="principle">
206     <div class="principle-title">Principle 1 · Resource-oriented REST</div>
207     <p>
208       Every YouTube entity — <strong>videos</strong>, <strong>channels</strong>, <strong>playlists</strong>,
209       <strong>comments</strong> — is a <em>resource</em> with a stable URL.
210       You navigate the graph by following resource IDs, not by calling procedural methods.
211       The <code>search.list</code> endpoint returns lightweight references; you then call
212       <code>videos.list</code> with those IDs to fetch full resource representations.
213     </p>
214   </div>
215 
216   <!-- ?? Principle 2 ?? -->
217   <div class="principle">
218     <div class="principle-title">Principle 2 · Partial responses (field masks)</div>
219     <p>
220       The API never forces you to receive the whole resource. The <code>part</code> parameter
221       selects top-level <em>resource parts</em> (<code>snippet</code>, <code>statistics</code>,
222       <code>contentDetails</code> …), and <code>fields</code> narrows down to exact properties.
223       This keeps payloads small and quota costs low — each part has an independent quota weight.
224     </p>
225   </div>
226 
227   <!-- ?? Principle 3 ?? -->
228   <div class="principle">
229     <div class="principle-title">Principle 3 · Cursor-based pagination</div>
230     <p>
231       Result sets are delivered in pages of up to <code>maxResults</code> items.
232       The response contains a <code>nextPageToken</code> (and optionally <code>prevPageToken</code>)
233       you pass back as <code>pageToken</code> on the next request.
234       This is stable under concurrent writes — unlike offset-based pagination.
235     </p>
236   </div>
237 
238   <!-- ?? Principle 4 ?? -->
239   <div class="principle">
240     <div class="principle-title">Principle 4 · API key + quota system</div>
241     <p>
242       All unauthenticated read access uses an <strong>API key</strong> attached as a query
243       parameter. Write operations and user-scoped data require OAuth 2.0.
244       Every method costs a fixed number of <em>quota units</em> per day (default: 10 000).
245       <code>search.list</code> costs 100 units; <code>videos.list</code> costs 1.
246       Design accordingly: cache aggressively, request only the parts you need.
247     </p>
248   </div>

324 // ?? Helpers ??????????????????????????????????????????????????????????????????
325 function iso8601ToHuman(d) {
326   // PT4M13S ? 4:13
327   const m = d.match(/PT(?:(\d+)H)?(?:(\d+)M)?(?:(\d+)S)?/);
328   if (!m) return d;
329   const h = parseInt(m[1]||0), mn = parseInt(m[2]||0), s = parseInt(m[3]||0);
330   if (h) return `${h}:${String(mn).padStart(2,'0')}:${String(s).padStart(2,'0')}`;
331   return `${mn}:${String(s).padStart(2,'0')}`;
332 }
333 function fmtNum(n) {
334   if (!n) return '—';
335   const v = parseInt(n);
336   if (v >= 1e6) return (v/1e6).toFixed(1)+'M';
337   if (v >= 1e3) return (v/1e3).toFixed(1)+'K';
338   return v.toString();
339 }

342 async function runDemo() {
343   const apiKey     = document.getElementById('apiKey').value.trim();
344   const query      = document.getElementById('query').value.trim();
345   const maxResults = document.getElementById('maxResults').value;
346   const order      = document.getElementById('order').value;
347   const out        = document.getElementById('output');
348   const btn        = document.getElementById('runBtn');
349 
350   if (!apiKey) { out.innerHTML = '<p class="error">? Paste your API key above.</p>'; return; }
351   if (!query)  { out.innerHTML = '<p class="error">? Enter a search query.</p>'; return; }
352 
353   btn.disabled = true;
354   out.innerHTML = '<p class="spinner">Fetching… (2 requests: search.list ? videos.list)</p>';
355 
356   try {
357     // ?? PRINCIPLE 1 + 3: search.list ? lightweight resource references + pagination cursor
358     const searchUrl = new URL("https://www.googleapis.com/youtube/v3/search");
359     searchUrl.searchParams.set("key",        apiKey);
360     searchUrl.searchParams.set("q",          query);
361     searchUrl.searchParams.set("type",       "video");
362     searchUrl.searchParams.set("part",       "id");           // PRINCIPLE 2: only id part
363     searchUrl.searchParams.set("maxResults", maxResults);
364     searchUrl.searchParams.set("order",      order);
365 
366     const searchRes  = await fetch(searchUrl);
367     const searchData = await searchRes.json();
368 
369     if (searchData.error) throw new Error(searchData.error.message);
370 
371     const videoIds = searchData.items.map(i => i.id.videoId).join(",");
372     const nextPage = searchData.nextPageToken;
373 
374     // ?? PRINCIPLE 1: resolve IDs ? full Video resources
375     // ?? PRINCIPLE 2: field mask keeps payload minimal
376     const videoUrl = new URL("https://www.googleapis.com/youtube/v3/videos");
377     videoUrl.searchParams.set("key",    apiKey);
378     videoUrl.searchParams.set("id",     videoIds);
379     videoUrl.searchParams.set("part",   "snippet,statistics,contentDetails");
380     videoUrl.searchParams.set("fields", "items(id,snippet(title,channelTitle,thumbnails/medium),statistics(viewCount,likeCount),contentDetails(duration))");
381 
382     const videoRes  = await fetch(videoUrl);
383     const videoData = await videoRes.json();
384     if (videoData.error) throw new Error(videoData.error.message);
385 
386     // ?? Render ??????????????????????????????????????????????????????????????
387     let html = `<h3>RESULTS · ${videoData.items.length} videos · quota used ? 101 units</h3>`;
388 
389     for (const v of videoData.items) {
390       const sn = v.snippet, st = v.statistics, cd = v.contentDetails;
391       html += `
392         <div class="result-item">
393           <img class="thumb" src="${sn.thumbnails?.medium?.url || ''}" alt="" loading="lazy">
394           <div class="result-info">
395             <h4>${sn.title}</h4>
396             <div class="meta">${sn.channelTitle} · ${iso8601ToHuman(cd.duration)} · ?? ${fmtNum(st.viewCount)} · ?? ${fmtNum(st.likeCount)}</div>
397             <a href="https://www.youtube.com/watch?v=${v.id}" target="_blank" rel="noopener">? watch on YouTube</a>
398           </div>
399         </div>`;
400     }
401 
402     if (nextPage) {
403       html += `<p style="margin-top:12px;font-size:0.78rem;color:var(--muted);font-family:'Space Mono',monospace">
404         PRINCIPLE 3 — cursor-based pagination: nextPageToken = <code style="color:var(--accent2)">${nextPage}</code><br>
405         Pass it as <code style="color:#aaa">pageToken=${nextPage}</code> to fetch the next page.
406       </p>`;
407     }
408 
409     // Raw JSON toggle
410     html += `<button class="raw-toggle" onclick="toggleRaw()">SHOW RAW JSON</button>
411       <pre id="rawJson" class="code-block" style="font-size:0.7rem;max-height:300px;overflow:auto">${JSON.stringify(videoData, null, 2)}</pre>`;
412 
413     out.innerHTML = html;
414 
415   } catch(e) {
416     out.innerHTML = `<p class="error">Error: ${e.message}</p>`;
417   } finally {
418     btn.disabled = false;
419   }
420 }
